# 风险拦截说明手册

**AgentWatch Runtime Gate — 运营者快速参考**

本手册面向绑定了 CLI Agent 的运营者。Runtime Gate 在每一次「工具调用」的边界做实时拦截——不是读完你的 prompt 再判断，而是在 Agent 真的要执行转账、删文件、运行脚本的那一瞬间检查参数。本文档覆盖 8 条确定性规则（L0）、3 条组合增强规则（Fusion）以及 1 套统计引擎兜底（L1），帮助你理解每一次 WARN/BLOCK 背后的真实威胁、为什么会触发、以及你应该怎么处理。

> **归档说明**：机器可读映射见 [RULE_USER_COPY.json](./RULE_USER_COPY.json)；Dashboard 实现见 `packages/web/src/lib/riskCopy.ts`。

## 一、拦截总览表

### 1.1 L0 规则引擎（确定性匹配）

L0 采用确定性匹配——参数满足条件即触发。所有规则在工具调用发生的瞬间完成判断。

| userTitleZh | ruleId | 动作 | 触发条件（白话） | 潜在危害 | 建议处置 |
|-------------|--------|------|------------------|----------|----------|
| 指令劫持 | GOAL_HIJACK_001 | BLOCK | 参数含 "ignore previous instruction" 等覆盖指令 | Agent 忽略原始任务，行为失控 | 暂停 Agent，检查输入来源 |
| 角色覆盖 | GOAL_HIJACK_002 | BLOCK | 角色设定语句、"you are now" 或 40 位 hex 地址 | 覆盖系统角色或伪装地址引导转账 | 检查上下文，调整输入格式 |
| 大额转账 | PARAM_TAMPER_001 | BLOCK | transfer 且 amount/value/sum ≥ 10 万 | 超阈值转账上链不可逆 | 人工复核金额与收款地址 |
| 工具链滥用 | CHAIN_ABUSE_001 | BLOCK | 敏感工具 + 链深度 ≥ 3 | 多步编排攻击（读密钥→外传→转账） | 逐步检查调用链 |
| 权限探测 | PERM_PROBE_001 | WARN | 连续 3 次及以上调用失败 | 系统性探测权限边界 | 观察失败是否呈规律轮换 |
| 供应链风险 | SUPPLY_CHAIN_001 | WARN | 工具来源不在白名单 | 未知来源可能含后门 | 仅允许经审计的工具来源 |
| 极端频率 | FREQ_001 | BLOCK | 1 分钟内调用 ≥ 100 次 | 自动化攻击或失控循环 | 暂停 Agent，确认批处理 |
| 分隔符注入 | PROMPT_INJ_001 | WARN | 连续 `---`/`###` 或 HTML 闭合标签 | 破坏 prompt 结构夹带指令 | 检查分隔符前后内容 |

### 1.2 组合增强规则（Fusion）

组合规则将多个 L0 场景得分融合，提升综合威胁等级。**Dashboard Live 暂未上云 `triggeredCombinations`，以下组合由 L0 命中集本地推断。**

| userTitleZh | combination id | 推断条件 | 含义 |
|-------------|----------------|----------|------|
| 高危转账组合 | high_value_transfer | PARAM_TAMPER_001 + CHAIN_ABUSE_001 | 大额 + 多步链，需双重复核 |
| 协同攻击组合 | coordinated_attack | PROMPT_INJ_001 + GOAL_HIJACK_* | 注入 + 劫持，典型协同攻击 |
| 快速试探组合 | rapid_probing | PERM_PROBE_001 + FREQ_001 | 高频 + 权限试探，疑似渗透 |

### 1.3 L1 统计引擎兜底

L0 未命中时，L1 对行为基线做多维异常检测：

| 维度 | 权重（架构设计） | 说明 |
|------|------------------|------|
| 工具链深度 | 0.25 | 链越深越可能编排攻击 |
| 工具频率 | 0.20 | 单位时间同工具频次 |
| 错误率 | 0.15 | 高失败率可能暗示探测 |
| 参数数量 | 0.15 | 单次参数个数异常 |
| 用户重复 | 0.15 | 相同来源重复模式 |
| 延迟 | 0.10 | 响应时间异常 |

融合：`combinedScore = zScore×0.35 + frequency×0.25 + markov×0.25 + cusum×0.075 + ewma×0.075`

- `combinedScore ≥ 0.7` → 异常告警
- Dashboard 仅展示 **综合分**，暂无各维度分项（Live）

**运营者须知**：无 L0 命中但 L1 分高 → 行为模式统计异常，需结合上下文人工判断。


## 四、FAQ

**Q1：为什么 transfer 被拦？**  
常见原因：金额 ≥ 10 万（PARAM_TAMPER_001）、链深度 ≥ 3 的高危链（CHAIN_ABUSE_001）、或 L1 统计异常。先查 `l0_triggered_rules`。

**Q2：WARN 和 BLOCK 区别？**  
WARN：工具已执行，标记可疑，需事后审查。BLOCK：工具未执行，必须先介入。

**Q3：Demo 和 Live 区别？**  
Demo 来自 mockData（仅供 UI）；Live 来自真实 Runtime Gate。若看到 `LARGE_TRANSFER_001` 说明是旧 Mock。

**Q4：分数 91 是什么意思？**  
即 `l1_combined_score = 0.91`，行为相对基线高度异常。有 L0 时优先看 L0；无 L0 时人工结合工具与时间线判断。

**Q5：误拦怎么办？**  
查触发规则 → 对照 falsePositive 场景 → 调整阈值/白名单/可信链，而非长期关规则。

**Q6：和「读 prompt 的安全产品」有何不同？**  
Prompt 安全在输入层；Runtime Gate 在 **tools/call 执行边界**——即使 prompt 层被绕过，仍可在危险操作前拦截。

**Q7：同一操作为何有时拦有时放？**  
L0 确定性；L1 看上下文（链深、频率、失败率等）。同一 transfer 在链深 1 vs 4 时结论可不同。

**Q8：组合规则 Dashboard 能直接看到吗？**  
Live 不能。同时出现 PARAM_TAMPER + CHAIN_ABUSE 时可推断 high_value_transfer。

**Q9：V0 已知缺口？**  
耗时异常弱覆盖；A2A/基线偏离可选；组合与 L1 分项未上云。

**Q10：0x + 40 位 hex 为何触发？**  
GOAL_HIJACK_002 防伪装地址嵌入参数；正常讨论合约地址时可能标记，需人工确认。
